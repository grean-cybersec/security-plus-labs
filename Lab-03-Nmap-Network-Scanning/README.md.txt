# Lab 03 — Nmap Network Scanning & Reconnaissance

## Objective
Use Nmap to perform host discovery and port scanning on a home network,
identify exposed services, and analyze the security posture of a router's
web administration panel — including how credentials are transmitted.

## Tools Used
- Nmap 7.99 (Windows 10)
- Wireshark 4.6.6
- PowerShell — ipconfig

## Network Environment
- Subnet: 192.168.100.0/24
- Devices identified: 5 hosts (router, laptop, phone, tablet, repeater)
- Both 2.4GHz and 5GHz bands bridged into the same subnet by the router

## Steps Performed

### 1. Host Discovery
Identified all live devices on the local network using a ping scan:
nmap -sn 192.168.100.0/24
Result: 5 active hosts found, including the gateway (router) and all
personal devices connected via WiFi (both frequency bands).

### 2. Port Scanning with Service Detection
Scanned the router for open ports and running services:
nmap -sV 192.168.100.1
Result: Port 80 (HTTP) open — router's web administration interface.
999 other common ports closed — minimal attack surface.

### 3. Credential Transmission Analysis
Captured the router login process with Wireshark to determine how
credentials are transmitted over the network.

## Key Findings

| Finding | Detail |
|---|---|
| Open ports on router | Only 80/tcp (HTTP) — no HTTPS available |
| Admin panel exposure | Login required, but served over unencrypted HTTP |
| Credential format | JSON (application/json), not standard form encoding |
| Credential protection | Client-side hashed before transmission — NOT plaintext |
| Security headers present | X-Frame-Options, X-XSS-Protection — basic protections active |

## Analysis — HTTP Transport vs Credential Protection

This lab revealed an important distinction often tested on the exam:

**Transport-layer encryption (HTTPS/TLS) vs application-layer protection.**

The router does NOT use HTTPS — all traffic is unencrypted at the
transport layer. However, it does NOT send the raw password either —
it sends a client-side hash generated by JavaScript before the request
is transmitted.

| | Lab 02 (custom server) | Lab 03 (real router) |
|---|---|---|
| Transport encryption | None (HTTP) | None (HTTP) |
| Credential format | Plaintext form field | Hashed value (JSON) |
| Wireshark visibility | Full password visible | Only hash visible |
| Security level | Critical risk | Partial mitigation |

**Important exam note:** client-side hashing is NOT equivalent to HTTPS.
It reduces immediate exposure but remains vulnerable to **replay attacks**
(an attacker could capture and resend the same hash to authenticate
without ever knowing the real password) and does not protect other
session data.

## Security Relevance
In a SOC analyst role, network reconnaissance is the foundation of:
- **Asset inventory** — knowing what's connected to the network
- **Attack surface mapping** — identifying exposed services
- **Vulnerability assessment** — flagging unencrypted admin panels

Routers with HTTP-only admin interfaces are a common finding in home
and small office network audits — frequently flagged in security
assessments as a Medium risk finding.

## Mitigation Recommendations
- Enable HTTPS for router administration if supported by firmware
- Change default router credentials immediately after setup
- Disable remote administration access from outside the local network
- Segment IoT devices onto a separate VLAN where possible
- Keep router firmware updated to patch known vulnerabilities

## Lessons Learned
- CIDR notation (/24) defines the scan range — not the WiFi band
- 2.4GHz and 5GHz typically share the same subnet on consumer routers
- `-sV` flag reveals service version — critical for vulnerability research
- Not all HTTP traffic exposes plaintext credentials — some applications
  apply partial protections like client-side hashing
- Private IP addresses (RFC 1918) are safe to document in security
  portfolios — they are not internet-routable or identifying
